## 2. 启动与运行

项目包含一键启动脚本，可同时开启后台服务与 Electron 客户端。

### 2.1 一键启动

在项目根目录下，直接执行：

```bash
cd /Users/ffm1110/workapp/rpa
chmod +x start.sh
./start.sh
```

此脚本会：
1. 启动后端 Python 服务（后台运行，默认监听 `8765` 端口）。
2. 等待后端 `/health` 健康检查就绪。
3. 自动在开发模式下启动 Electron 前端客户端。
4. 当前端退出时，自动释放并终止后台 Python 进程。

### 2.2 手动分步启动

**第一步：启动 Python 服务**
```bash
cd /Users/ffm1110/workapp/rpa
source .venv/bin/activate
python rpa_core/server.py
```

**第二步：启动 Electron 客户端**
```bash
cd /Users/ffm1110/workapp/rpa/rpa-client
npm run electron:dev
```

## 3. API 鉴权与安全机制

后端服务内置了 **API Token 鉴权保护**，防止跨域非法调用。

### 3.1 获取 Token

1. **自动生成**：若未配置环境变量，启动 `server.py` 时会自动生成一个随机 UUID，并在 stdout 打印：
   ```
   __RPA_API_TOKEN_START__=xxxx-xxxx-xxxx-xxxx
   ```
   Electron 客户端启动时会自动解析该值并建立信任连接。
2. **手动指定**：也可以在启动时注入环境变量：
   ```bash
   RPA_API_TOKEN="my_custom_secure_token" python rpa_core/server.py
   ```

### 3.2 发送请求认证

向接口发送请求时，必须在 HTTP 标头中携带 Token：
- 标头名：`X-RPA-Token` 或以 `Authorization: Bearer <Token>` 形式提供。

示例：
```bash
curl -H "X-RPA-Token: 你的_TOKEN" http://127.0.0.1:8765/health
```

## 4. WebSocket 实时执行与远程控制

为保证大批步骤执行不阻塞前端并实现实时流式日志传输，项目引入了 WebSocket 双向通信通道 `/ws/execute`。

### 4.1 握手与连接

- **地址**：`ws://127.0.0.1:8765/ws/execute?token=你的_TOKEN`

### 4.2 消息协议

#### (1) 客户端发起运行 (start)
连接建立后，客户端必须发送动作 `start` 和流程定义 JSON：
```json
{
  "action": "start",
  "flow": {
    "name": "MyFlow",
    "steps": [...]
  },
  "initial_context": {
    "variable_name": "value"
  }
}
```

#### (2) 服务端推送实时状态 (step_start / step_end / status / result)
- **步骤开始**：`{"type": "step_start", "data": {"step_index": 0, "step_type": "open"}}`
- **步骤结束**：`{"type": "step_end", "data": {"step_index": 0, "success": true, "duration_ms": 1500, "message": "..."}}`
- **运行状态**：`{"type": "status", "data": {"status": "paused|running", "message": "..."}}`
- **总执行结果**：`{"type": "result", "data": {"success": true, "executed_steps": 5, "total_steps": 5, "context": {...}}}`

#### (3) 客户端下发实时控制指令
执行过程中，客户端可向 WebSocket 随时发送控制消息：
- **暂停**：`{"action": "pause"}`
- **恢复**：`{"action": "resume"}`
- **停止**：`{"action": "stop"}`

## 5. Flow 流程定义 (嵌套控制流)

Flow 是支持多级嵌套控制流的 JSON 文件。

### 5.1 基本骨架

```json
{
  "id": "flow_unique_id",
  "name": "示例流程",
  "description": "说明描述",
  "steps": [
    {
      "id": "step_1",
      "type": "open",
      "value": "https://example.com"
    },
    {
      "id": "step_2",
      "type": "if",
      "condition": "{{my_var}} == 'success'",
      "then": [
        { "id": "step_3", "type": "click", "selector": "#ok-btn" }
      ],
      "else": [
        { "id": "step_4", "type": "click", "selector": "#cancel-btn" }
      ]
    }
  ]
}
```

### 5.2 变量 Context

在步骤的参数中填入 `{{变量名}}` 可以在运行时获取动态值。
- 支持类型保留：若上下文中的变量是列表，如 `my_list = [1, 2, 3]`，插值 `{{my_list}}` 将作为原始数组类型传入后续步骤，而非直接被强制转为字符串。

## 6. Step 类型参考

### 6.1 基础步骤

#### open — 打开页面
```json
{ "type": "open", "value": "https://www.baidu.com", "timeout": 15, "on_fail": "abort" }
```
- `value`: (必填) 目标网页地址。
- `timeout`: (选填) 页面载入超时时间（秒），默认 10。

#### click — 点击元素
```json
{ "type": "click", "selector": "#submit-btn", "timeout": 10 }
```
- `selector`: (必填) 元素的 CSS 选择器（如 `#id`、`.class`）。

#### input — 输入文本
```json
{ "type": "input", "selector": "#search-input", "value": "RPA 自动化", "timeout": 10 }
```
- `selector`: (必填) 目标输入框选择器。
- `value`: (必填) 要键入的文本。

#### wait — 等待延迟
```json
{ "type": "wait", "value": "3" }
```
- `value`: (必填) 延迟等待时间（秒）。

#### extract — 提取元素内容
```json
{
  "type": "extract",
  "selector": "h1.title",
  "context_key": "page_title",
  "save_path": "output/title.txt"
}
```
- `selector`: (必填) 目标文字所在节点。
- `context_key`: (选填) 保存到运行时上下文中的变量名，可用于后续的 If/Loop 判断。
- `save_path`: (选填) 输出保存到本地磁盘的文件路径。

---

### 6.2 流程控制步骤

#### if — 条件判断分支
```json
{
  "type": "if",
  "condition": "{{status}} == 'success'",
  "then": [
    { "type": "click", "selector": "#next" }
  ],
  "else": [
    { "type": "open", "value": "https://alternative.com" }
  ]
}
```
- `condition`: (必填) 表达式字符串。支持 `==`、`!=`、`>`、`<`、`>=`、`<=` 以及 `contains`（包含子串检查）。

#### loop — 循环重复执行
```json
{
  "type": "loop",
  "loop_type": "each",
  "value": "{{target_links}}",
  "item_key": "link_item",
  "index_key": "link_index",
  "steps": [
    { "type": "open", "value": "{{link_item}}" }
  ]
}
```
- `loop_type`: (必填) 循环模式：
  - `count`：按次数循环。`value` 为具体数值（如 `5`）或代表数值的变量。
  - `each`：遍历数组/列表。`value` 为指向数组的变量（如 `{{my_array}}`）。
- `item_key`: (选填) 当前循环项所绑定的临时上下文键名，默认值为 `item`。
- `index_key`: (选填) 当前循环索引（从 0 开始）所绑定的上下文键名，默认值为 `index`。
- `steps`: (必填) 循环体步骤列表。

---

### 6.3 失败重试策略 (on_fail: retry)

对于可能存在动态波动的外部步骤，失败处理策略（`on_fail`）除了支持 `abort` (直接退出) 和 `skip` (跳过)，现在增加了 **`retry` (重试机制)**。

```json
{
  "type": "click",
  "selector": ".dynamic-btn",
  "on_fail": "retry",
  "max_retries": 3,
  "retry_delay": 2
}
```
- `on_fail`: 设置为 `retry` 以启动重试机制。
- `max_retries`: (选填) 最大重试次数，默认 3。
- `retry_delay`: (选填) 每次失败后间隔重试的等待时间（秒），默认 2。

## 7. 浏览器元素拾取指南

RPA 客户端支持 **可视化元素定位拾取**。

1. **启动浏览器**：在画布流程中首先配置并执行一个 `open` 步骤，以打开一个受接管的 Chrome 页面。
2. **选择目标步骤**：点击一个需要选择器参数的步骤（例如 `click`、`input`、`extract`）。
3. **激活拾取**：在右侧属性面板的“选择器”输入框旁，点击 **🎯 拾取** 按钮。客户端状态将切换为“拾取中”。
4. **选择网页元素**：切换到受 RPA 接管的浏览器页面，你会看到鼠标悬停的元素被高亮显示。
5. **确认拾取**：在网页上直接左键点击该元素。客户端会自动回传一组**按稳定性排序的候选选择器**（id > data-testid/name/aria-label > CSS 路径 > 文本），首选填入选择器输入框，其余作为候选列表显示。运行时若首选失效，引擎会自动按候选顺序回退（**选择器自愈**），并在日志里给出回退提示。
6. **手动调整候选**：在属性面板的候选列表里可增删候选、用 ▲ 调整优先级。

## 9. 常见问题

### Q1：一键启动报错：端口占用 (Address already in use)

前端或后端非正常退出时可能没有完全杀死端口。可以使用如下命令清理：
```bash
lsof -ti:8765 | xargs kill -9
```

### Q2：运行 if/loop 条件判断时类型报错 (TypeError)

- 确保插值变量在上下文中存在。
- 比较字符串时，建议用英文单引号/双引号包裹，如 `{{result}} == 'ok'`。
- 遍历列表前，请确保被遍历的上下文键值是通过 `extract` 成功获取并序列化出来的有效数组（或通过初始化传入）。

### Q3：浏览器启动闪退或报错

- 接管模式会检查当前系统是否有正在运行的相同调试端口的 Chrome 进程。如有，请先关闭其他 Chrome 浏览器或清理 Chrome 后台进程。
- 检查 Chrome 是否已经安装于默认目录 `/Applications/Google Chrome.app`。

### Q4：重新打开客户端后找不到之前的流程 / 如何管理多个流程

- 流程在你点击「保存」后会写入磁盘（见 [第 16 节](#16-数据存储位置)），不会丢失。
- **命名**：顶部的流程名是可编辑输入框，点一下直接改名，再「保存」即生效（不要让多个流程都叫"新流程"）。
- **打开**：点「**打开 ▾**」看到所有已保存流程（名称/时间/步数），点击打开；客户端启动时也会自动打开最近一个。
- **删除**：在「打开 ▾」列表里把鼠标移到某项上，点 🗑 删除。
- **导入/导出**：「导出」把当前流程存成 `.json` 文件（默认按流程名命名）；「导入」从文件选择一个 `.json` 流程载入（会赋新 id，不覆盖原有）。可用于备份或在机器间迁移。

## 10. 新增 Step 类型（v0.3）

### select — 下拉框选择
```json
{ "type": "select", "selector": "#city", "by": "text", "value": "北京" }
```
- `by`: 选择方式，`text`（可见文本）/ `value`（value 属性）/ `index`（下标，从 0 开始），默认 `text`。
- `value`: 要选择的选项。

### switch_tab — 切换 / 新建标签页
```json
{ "type": "switch_tab", "value": "latest" }
{ "type": "switch_tab", "new_tab": true, "url": "https://example.com" }
```
- `value`: `latest`（最新标签页）或下标（如 `0`、`1`）。
- `new_tab`: 为 `true` 时打开新标签页并切换过去，可选 `url`。
- 切换后，后续所有步骤都作用于该标签页。

### download — 文件下载
```json
{ "type": "download", "selector": "#export-btn", "save_path": "downloads", "timeout": 30 }
```
- `selector`: 触发下载的元素。
- `save_path`: 保存目录（受输出沙箱约束，见 [第 16 节](#16-数据存储位置)），默认 `downloads`。
- `timeout`: 等待下载完成的超时秒数。

## 11. 选择器进阶

### 11.1 多种定位方式
选择器支持前缀，无前缀默认按 **CSS** 解析：

| 写法 | 含义 |
|------|------|
| `#id` / `.class` / `div > a` | CSS（默认） |
| `xpath://div[@id='x']` 或 `//div...` | XPath |
| `text:登录` | 文本包含匹配 |
| `tag:button` / `@name=user` | 标签名 / 属性 |

### 11.2 候选选择器与自愈
步骤可带 `selectors`（候选列表，按优先级排序）；运行时依次尝试，命中即用，首选失效会自动回退：
```json
{ "type": "click", "selectors": ["#submit", "css:.btn-primary", "text:提交"] }
```

### 11.3 iframe 内定位
`click` / `input` / `extract` / `select` / `download` 可加 `frame` 在 iframe 内查找：
```json
{ "type": "input", "frame": "#editor-iframe", "selector": "body", "value": "正文" }
```

## 12. 运行历史

每次执行都会记录到 SQLite（`rpa_core/data/history.db`），含结果摘要与完整步骤日志；**步骤失败时自动截图**，路径写入对应日志条目。

- `GET /runs?limit=50` — 最近运行列表（摘要）
- `GET /runs/{run_id}` — 单次运行详情（含步骤日志与失败截图路径）

失败截图保存在 `rpa_core/output/runs/<run_id>/`。

## 13. 定时调度

后端内置调度引擎（cron / interval），定时执行与手动执行**默认串行**（共用一个可见浏览器、可观察、与拾取共用窗口）。

> **并发执行（可选）**：设 `RPA_BROWSER_POOL_SIZE=N`（N>1）启用 headless 浏览器池，手动与定时流程最多 N 个并发；池满时新请求返回 409 / WS error。默认 1，行为完全不变。

| 接口 | 说明 |
|------|------|
| `POST /schedules` | 创建任务 |
| `GET /schedules` | 列出任务 |
| `GET /schedules/{id}` | 任务详情 |
| `PATCH /schedules/{id}?enabled=false` | 启用/停用 |
| `DELETE /schedules/{id}` | 删除 |
| `POST /schedules/{id}/run-now` | 立即触发一次 |

创建示例（每天 9:00 执行）：
```json
{
  "name": "每日导出",
  "flow": { "name": "export", "steps": [ ... ] },
  "schedule_type": "cron",
  "schedule_value": "0 9 * * *"
}
```
- `schedule_type`: `cron`（5 字段：分 时 日 月 周，支持 `*` `*/n` `a-b` 列表）或 `interval`（秒数）。

## 14. 凭据保险库

账号密码等敏感值加密存储（Fernet/AES），明文不进入流程定义、运行历史或日志。

- 在客户端顶部点 **🔐 凭据** 管理；或用接口：
  - `POST /secrets` `{ "name": "...", "value": "..." }` — 写入（明文不回读）
  - `GET /secrets` — 仅返回名称
  - `DELETE /secrets/{name}` — 删除
- 在流程任意值字段用 `{{secret:名称}}` 引用：
```json
{ "type": "input", "selector": "#pwd", "value": "{{secret:login_pw}}" }
```
- 密钥来源：环境变量 `RPA_SECRET_KEY`，或自动生成的 `rpa_core/data/secret.key`（建议生产环境用环境变量注入外部托管的密钥）。

## 15. AI 能力（生成流程 + 视觉兜底）

本项目是**个人版**：AI 可选，要用就**在客户端里自配 API**（用你自己的 key）。两项能力都走官方 Claude（默认 `claude-opus-4-8`）。不配 AI 时，所有功能照常，只是没有 AI 生成与视觉兜底。

### 15.0 配置 AI（在客户端里）

点顶部 **「⚙️ AI 设置」**，填：
- **API Key**：你的 key，**加密存储、不回显**（存在加密保险库里，不会出现在凭据列表）。
- **API 地址（base_url）**：可选，默认官方；可填代理或自托管/兼容端点。
- **模型**：默认 `claude-opus-4-8`。
- **执行时开启 AI 视觉兜底**：勾选后，DOM 选择器全失效时才会调用 AI。

点「保存并测试」会做一次最小调用验证连通。也可用接口：`GET/POST /ai/config`、`POST /ai/config/test`。

> 高级/开发用法：也支持环境变量 `ANTHROPIC_API_KEY` / `ANTHROPIC_BASE_URL` / `RPA_LLM_MODEL` / `RPA_AI_FALLBACK`（客户端里的配置优先）。

### 15.1 自然语言生成流程

点客户端顶部 **「✨ AI 生成」**，用一句话描述需求（可填起始网址），AI 生成可编辑的 Flow 载入画布。也可用接口：
```bash
POST /flows/generate   {"instruction": "打开百度搜索 RPA 并提取首条标题", "url_hint": "https://www.baidu.com"}
# -> {"success": true, "flow": {...}}  生成结果会经本地校验，非法则返回 errors
```
- 只需在「⚙️ AI 设置」里配好 key（生成不依赖视觉兜底开关）。
- 生成后**务必检查选择器**——AI 的选择器是合理猜测，配合「目标描述」字段 + 视觉兜底更稳。

### 15.2 视觉兜底定位（选择器自动修复）

当一个步骤的 DOM 选择器（含候选列表）**全部失效**时，引擎可调用多模态 LLM 兜底：把页面截图 + 精简 DOM + 失效的选择器交给模型，让它返回一个**修好的选择器**，或在 DOM 无法定位时（canvas / 图形界面）返回**点击坐标**。这是选择器自愈的终局。

在「⚙️ AI 设置」里**勾选「执行时开启 AI 视觉兜底」**即生效（需已配 key）。未勾选或未配 key 时，引擎行为完全不变（纯 DOM 自愈）。

#### 用法
- 给步骤填一个**目标描述**（属性面板「目标描述」字段，或 JSON 的 `description`），如"登录按钮"，能显著提升兜底命中率。
- `click` 支持选择器修复与坐标点击；`input`/`extract`/`select` 仅用修复后的选择器（坐标无法用于这些操作）。
- 兜底命中会在日志里标注「AI 选择器修复」或「AI 视觉兜底」。

```json
{ "type": "click", "selector": "#login", "description": "页面右上角的登录按钮" }
```

## 17. 打包发布（独立可运行）

客户端 = Electron 前端 + Python 后端。打包时用 **PyInstaller** 把后端冻结成单文件可执行程序，再由 **electron-builder** 一并装进安装包，**目标机无需安装 Python**。

### 17.1 一键打包

在 `rpa-client/` 下，于**对应的目标系统**运行：
```bash
cd rpa-client
npm install                 # 首次
npm run package:mac         # 在 macOS 上 → 产出 .dmg / .zip
npm run package:win         # 在 Windows 上 → 产出 NSIS 安装包 / 免安装 exe
```
产物在 `rpa-client/release/`。脚本会先 `python ../build_backend.py` 冻结后端，再 `electron-builder` 打包。

### 17.2 两条硬约束

1. **PyInstaller 只能产出运行它的那个系统的可执行文件**——要同时有 Windows 和 macOS 版本，必须分别在 Windows 和 macOS 上各打一次（或用双 runner 的 CI）。不能在 Mac 上打出 Windows 版。
2. **目标机仍需安装 Google Chrome**——DrissionPage 驱动系统 Chrome，不随包打入（否则体积过大）。

### 17.3 注意事项
- 冻结后端的 **CPU 架构 = 打包所用 Python 的架构**。Apple Silicon 上若用 x86_64 的 Python，产出的后端是 x86_64（在 arm64 上靠 Rosetta 运行）。要纯 arm64 包，请用 arm64 的 Python 再打。
- 未配置代码签名时 electron-builder 用临时签名打包；正式分发建议配置开发者证书（macOS 公证 / Windows 签名）以免被系统拦截。
- 仅冻结后端做联调：`python build_backend.py` 后设 `RPA_BACKEND_BIN=<产物路径>` 再 `npm run electron:dev`，前端即用冻结后端而非本地 Python。

### 17.4 GitHub Actions 自动打包

仓库已带 `.github/workflows/build.yml`：macOS(arm64) / macOS(x64) / Windows(x64) 三个 runner 各自冻结后端 + 打包。

- **触发**：在 Actions 页手动运行，或推送 `v*` 标签（如 `git tag v0.4.0 && git push --tags`）。
- 推标签时，三平台产物（.dmg / .zip / .exe）会自动附加到对应的 **GitHub Release**。
- 这样 Windows 版也不用自己找机器——交给 windows runner 打。
- 前提：仓库已推到 GitHub（本地 `git init` 的仓库需先 `git remote add` 并 push）。
